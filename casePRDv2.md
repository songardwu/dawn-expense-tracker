# casePRDv2.md - 【一品沐：案場損益管理系統】

## 1. 專案背景與目標

將案場損益管理數位化。建立一套具備嚴謹財務邏輯、權限隔離與修改追蹤功能的管理工具，協助設計師與管理者精確掌握收款、成本與分潤獎金。

---

## 2. 使用者角色與權限

| 角色 | 查看權限 | 操作權限 |
|------|---------|---------|
| 設計師 | 僅限本人負責案場 | 1. 基礎資料與施工日期錄入<br>2. 收入、成本明細、折讓錄入<br>3. 登錄收款（待管理者確認） |
| 管理者 | 全公司所有案場 | 1. 獎金核對勾選 + 確認出帳<br>2. 設定各案場獨立分潤比<br>3. 結案鎖定/解鎖<br>4. 收款對帳確認<br>5. 成本科目管理 |

## 3. 功能區塊詳細規格

### 第一區塊：案場基本資料 (Project Profile)

- **欄位**：案名、業主姓名、電話、地址、簽約日期
- **案場編號**：手動輸入案名，系統自動產生流水號（如 `CASE-20260416-001`）作為內部識別碼
- **負責設計師**：一對一關聯（一個案場一位設計師）
- **施工日期**：日期區間選擇器（自動計算施工總天數）
- **案場狀態**：`進行中` → `已完工`（設計師標記）→ `結案`（管理者鎖定）

### 第二區塊：合約與收款進度 (Revenue Control)

- **收入組成**：系統家具金額 + 非系統家具金額 = 原簽約總金額

- **調整項**：
  - **追加減明細**：多筆記錄（日期 + 說明 + 金額），合計顯示淨追加減
  - **5% 營業稅**：手動輸入（代收代付性質）
  - **折讓/扣抵明細**：多筆記錄（項目名稱 + 金額），如丈量費、客變費等
  - **裝修押金**：獨立紀錄
    - `deposit_amount`：原始押金金額
    - `deposit_refund`：實際退還金額（預設 = 原始金額，可修改）
    - `deposit_status`：`待退` / `部分退還` / `已退`
    - 押金扣抵 = deposit_amount - deposit_refund，計入利潤

- **結算總價（最終應收金額）**：
  - 公式：`原簽約總金額 + 追加減淨額 + 5% 營業稅 - 折讓/扣抵合計`
  - 註：押金不納入結算總價（獨立追蹤線）

- **收款明細（實收進度）**：
  - 欄位：收款日期、付款方式（`現金` / `匯款` / `其他`）、實收金額
  - **對帳確認機制**：
    - 設計師登錄收款 → 狀態「待確認」
    - 管理者對帳勾選 → 狀態「已確認」
    - 僅「已確認」收款計入累計實收金額
  - 自動彙整：累計實收金額、剩餘尾款（結算總價 - 累計實收）

### 第三 & 四區塊：支出明細彙整 (Expense)

- **成本科目**：可擴充清單（管理者可新增科目，無需改程式碼）
- **每個科目記錄單一合計金額**（未來可擴充至多筆明細）

- **系統工程成本（小計 A）**：
  - 預設科目：系統板材、運費、組裝工資、五金(博士家)、3D製圖費、其他系統費用

- **非系統工程成本（小計 B）**：
  - 預設科目：室內裝修審查費、保護、木工、油漆、水電、木地板、泥作、拆除、其他非系統費用

### 第五區塊：損益分析 Dashboard (Live Analytics)

- **案場總成本**：小計 A + 小計 B

- **案場利潤**：
  - 公式：`(原簽約總金額 + 追加減淨額 + 折讓/扣抵合計 + 押金扣抵) - 案場總成本`
  - 註：5% 稅金為代收代付，不計入利潤；折讓/扣抵金額因先前已收取，需計入收入計算；押金扣抵（客戶放棄部分）計入收入

- **押金資訊**：獨立顯示押金收入、實際退還、扣抵金額

- **視覺化**：利潤為負數時，數值顯示為鮮紅色

### 第六區塊：結算與分潤 (Settlement)

- **設計師分潤比 (%)**：由管理者針對個別案場手動設定

- **設計師獎金**：案場利潤 × 分潤比 (%)
  - 利潤為負時，獎金仍按比例計算並顯示紅字

- **公司收益**：案場利潤 - 獎金金額

- **獎金出帳流程**：
  1. 管理者勾選「獎金核對」
  2. 啟用「確認出帳」按鈕
  3. 點擊後產生一筆報帳紀錄（一案一筆，一次出帳）
  4. 負數獎金：按鈕仍可用，由管理者自行決定是否出帳

- **報帳欄位對應**：
  | reports 欄位 | 對應值 |
  |---|---|
  | vendor | 設計師姓名 |
  | category | 固定「設計師獎金」 |
  | project_no | 案場編號/案名 |
  | amount | 計算出的獎金金額 |

- **出帳後差異警示**：
  - 出帳後若案場資料被修改導致獎金變動，Dashboard 顯示警示
  - 格式：「已出帳 XX / 目前應為 YY / 差異 ZZ」
  - 由管理者決定如何處理

## 4. 系統核心機制

- **修改紀錄 (Audit Log)**：
  - 所有金額更動（合約、稅金、折讓、收款、成本）
  - 所有狀態變更（收款確認、押金狀態、案場狀態、結案鎖定/解鎖）
  - 記錄：修改人、時間、舊值、新值
  - 解鎖需輸入原因並存入日誌

- **管理者核對**：人工勾選獎金核對標記 + 確認出帳按鈕（兩步驟）

- **結案鎖定**：鎖定後全案進入唯讀模式。解鎖需輸入原因並存入日誌

## 5. 驗收標準 (AC)

| 編號 | 驗收項目 | 說明 |
|------|---------|------|
| AC1 | 裝修審查費測試 | 在成本區輸入審查費後，利潤與獎金同步扣除對應金額 |
| AC2 | 折讓與應收測試 | 折讓/扣抵減少尾款，但正確計入利潤 |
| AC3 | 權限測試 | 設計師無法編輯分潤比及他人案場 |
| AC4 | 負數利潤測試 | 利潤為負時獎金仍按比例計算、顯示紅字，管理者可選擇不出帳 |
| AC5 | 收款對帳測試 | 未確認的收款不計入累計實收和尾款 |
| AC6 | 押金扣抵測試 | 部分退還時，扣抵金額正確計入利潤 |
| AC7 | 獎金出帳測試 | 確認出帳後，reports 產生正確的報帳紀錄（vendor/category/project_no/amount） |
| AC8 | 負數獎金出帳測試 | 負數獎金案場，管理者不按出帳則不產生紀錄 |
| AC9 | 出帳後差異警示測試 | 已出帳後修改成本，系統顯示差異提醒 |

## 7. 與現有系統的整合

- 案場損益模組為**獨立模組**
- **唯一整合點**：獎金出帳時，產生一筆報帳紀錄至現有 reports 系統
- 現有報帳功能零改動

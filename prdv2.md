# 出帳管理系統 V2：帳號管理 + 權限控管

**版本:** v2.0
**前置條件:** MVP (v1.0) 已完成並部署到 Vercel + Postgres,主線功能(款項提報 + Excel 報表匯出)跑得通
**目標:** 讓多位設計師各自提報、互不可見,管理員可以看全部、改全部、匯出全部

---

## 1. 這份文件要解決什麼

MVP 階段只有 Dawn 一個人用,所以沒有做登入跟權限。但實際場景是:

- 公司有 3-5 位設計師,每個人各自跑不同的室內設計案
- 每個設計師要自己提報自己案場的廠商款項
- 設計師之間不應該看到彼此的提報內容(牽涉案場金額、廠商報價,是商業敏感資訊)
- 管理員(Dawn 跟合夥人)要能看到所有人的提報,做核對跟匯出

這份 prd_v2.md 要在 MVP 的基礎上加三件事:

1. **帳號系統** — 讓設計師能登入
2. **資料隔離** — 設計師只看得到自己的提報
3. **管理員全域權限** — 管理員看得到全部、可以匯出全部

---

## 2. 使用者角色定義

| 角色 | 說明 | 能做什麼 | 不能做什麼 |
|------|------|----------|-----------|
| 設計師 | 第一線的室內設計師,負責各自的案場 | 提報自己的款項、查看自己的提報清單、刪除自己的提報 | 看別人的提報、匯出全部報表、管理帳號 |
| 管理員 | Dawn 跟合夥人 | 查看全部提報、刪除任何人的提報、匯出全部報表(含年度累計)、管理帳號(新增 / 停用設計師) | 無限制 |

**關鍵規則:設計師 A 絕對不能透過任何方式(URL 直打、API 呼叫、瀏覽器開發者工具)看到設計師 B 的資料。這不是前端隱藏,是後端過濾。**

---

## 3. 帳號管理功能

### 3.1 帳號建立(管理員操作)

管理員在後台有一個「帳號管理」頁面,可以:

- **新增帳號**:輸入姓名、帳號名稱(英文,例如 `designer_a`)、預設密碼、角色(設計師/管理員)
- **停用帳號**:不做刪除,做「停用」。停用的帳號無法登入,但歷史提報資料保留(因為那些款項已經在帳上了,不能跟著人消失)
- **重設密碼**:管理員可以幫設計師重設回預設密碼

**不做的:**
- ❌ 設計師自己註冊(帳號由管理員統一建立)
- ❌ 忘記密碼 / email 驗證 / OTP(公司內部系統,管理員直接重設)
- ❌ 大頭照、個人資料頁(不需要)

### 3.2 登入頁面

打開系統首先看到登入頁面。輸入帳號 + 密碼,驗證成功後進入系統。

**技術細節:**
- 密碼用 bcrypt 雜湊儲存,**絕對不能明碼**
- 登入後用 Flask session(server-side session)或 JWT token 維持狀態
- Session 過期時間建議 7 天(Dawn 不想每天重新登入)
- 登入失敗顯示「帳號或密碼錯誤」,不要分開提示(安全考量)

### 3.3 登出

頁面右上角永遠顯示目前登入的使用者名稱 + 「登出」按鈕。點登出,清除 session,跳回登入頁。

## 4. 資料庫 schema 變更

在 MVP 的 `reports` 表基礎上,新增一張 `users` 表,並在 `reports` 表加一個 `user_id` 欄位。

### 4.1 新增 users 表

```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  username VARCHAR(50) UNIQUE NOT NULL,
  display_name VARCHAR(100) NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  role VARCHAR(20) NOT NULL CHECK (role IN ('designer', 'admin')),
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.2 修改 reports 表

```sql
ALTER TABLE reports ADD COLUMN user_id INTEGER REFERENCES users(id);
```

**資料遷移:** MVP 階段已經存在的提報沒有 user_id,統一歸屬給 Dawn 的管理員帳號:

```sql
-- 假設 Dawn 的 user_id = 1
UPDATE reports SET user_id = 1 WHERE user_id IS NULL;
-- 之後加 NOT NULL constraint
ALTER TABLE reports ALTER COLUMN user_id SET NOT NULL;
```

## 5. 權限控管(最重要的一節)

### 5.1 後端過濾規則(API 層)

每一個涉及 `reports` 資料的 API,都必須根據登入使用者的角色做過濾:

**設計師:**
```python
# 設計師只能看/操作自己的
@app.route('/api/reports')
def list_reports():
    user = get_current_user()
    if user.role == 'designer':
        reports = db.query("SELECT * FROM reports WHERE user_id = %s", user.id)
    elif user.role == 'admin':
        reports = db.query("SELECT * FROM reports")  # 管理員看全部
    return jsonify(reports)
```

**刪除:**
```python
@app.route('/api/reports/<int:id>', methods=['DELETE'])
def delete_report(id):
    user = get_current_user()
    report = db.query("SELECT * FROM reports WHERE id = %s", id)
    
    if user.role == 'designer' and report.user_id != user.id:
        return jsonify({"error": "無權限刪除他人的提報"}), 403
    
    db.execute("DELETE FROM reports WHERE id = %s", id)
    return jsonify({"ok": True})
```

### 5.2 驗收測試案例(一定要跑)

| # | 測試步驟 | 預期結果 |
|---|----------|----------|
| 1 | 用設計師 A 帳號登入,提報 3 筆 | 清單顯示 3 筆 |
| 2 | 登出,用設計師 B 帳號登入 | 清單顯示 0 筆(空的) |
| 3 | 設計師 B 提報 2 筆 | 清單顯示 2 筆(只有自己的) |
| 4 | 設計師 B 嘗試在瀏覽器 URL 直接打 `/api/reports/1`(這是設計師 A 的那筆) | 系統回傳 403 或 404,不顯示資料 |
| 5 | 設計師 B 嘗試刪除 id=1 | 系統回傳 403,不讓刪 |
| 6 | 登出,用管理員帳號登入 | 清單顯示 5 筆(全部人的) |
| 7 | 管理員匯出 Excel | Excel 包含全部 5 筆,有「提報人」欄位 |
| 8 | 管理員刪除設計師 A 的某筆 | 成功,清單變 4 筆 |

**第 4 跟第 5 步是最重要的——這就是 Dawn 原版 PRD 第二條驗收標準「設計師 A 嘗試透過 URL 或任何方式讀取設計師 B 的資料應被系統擋下」的具體實現。**

### 5.3 前端差異

**設計師看到的畫面:**
- 清單頁:只看到自己的提報
- 新增提報:跟 MVP 一樣
- 沒有「帳號管理」入口
- 沒有「匯出全部報表」按鈕(只能匯出自己的)

**管理員看到的畫面:**
- 清單頁:看到所有人的提報,每筆多一個「提報人」欄位
- 新增提報:跟 MVP 一樣(管理員自己也可以提報)
- 多一個「帳號管理」入口(側邊欄或頂端選單)
- 「匯出 Excel」匯出全部人的資料,Excel 多一個「提報人」欄位

## 6. Excel 報表變更

V2 的 Excel 在 MVP 的基礎上加一個欄位:

**明細頁籤:** 新增「提報人」欄位,排在最前面

| 提報人 | 日期 | 廠商類型 | 廠商名稱 | 案號 | 階段 | 款項分類 | 金額 | 憑證編號 |
|--------|------|----------|----------|------|------|----------|------|----------|
| 設計師 A | 2026-04-01 | 水電 | 好工地 | S001 | 水電工程 | 案場成本 | 35,000 | — |

**總覽頁籤:** 多一個「按提報人」的彙總區塊:

```
=== 按提報人彙總 ===
設計師 A: 案場成本 120,000 / 管銷 15,000 / 獎金 0 / 小計 135,000
設計師 B: 案場成本 80,000 / 管銷 8,000 / 獎金 20,000 / 小計 108,000
合計: 243,000
```

**設計師匯出:**
- 設計師只能匯出自己的資料
- Excel 不含「提報人」欄位(因為全部都是自己的,多餘)
- 總覽頁籤不含「按提報人彙總」(只有自己,沒意義)

## 7. 帳號管理頁面(管理員專用)

管理員在側邊欄或頂端選單有一個「帳號管理」入口,進去看到:

**帳號清單:**

| 帳號 | 姓名 | 角色 | 狀態 | 操作 |
|------|------|------|------|------|
| designer_a | 設計師 A | 設計師 | 啟用中 | [重設密碼] [停用] |
| designer_b | 設計師 B | 設計師 | 啟用中 | [重設密碼] [停用] |
| dawn | Dawn | 管理員 | 啟用中 | — |

**新增帳號按鈕:** 打開表單,填姓名、帳號、預設密碼、角色(下拉:設計師/管理員),送出。

**停用帳號:** 點「停用」按鈕,帳號變成灰色、無法登入,但歷史提報資料保留在報表裡。可以隨時「重新啟用」。

**重設密碼:** 點「重設密碼」,系統把密碼改回預設值(或管理員指定新密碼),下次設計師登入時用新密碼。

## 8. 刻意不做的事

| 不做的東西 | 為什麼不做 |
|-----------|-----------|
| 設計師自己註冊 | 公司內部系統,帳號由管理員控制 |
| 忘記密碼 / email 驗證 | 直接找管理員重設,省一塊工 |
| 密碼複雜度檢查 | 公司內部,先不強制 |
| 修改密碼頁面(設計師自己改) | V3 再做,目前管理員重設就好 |
| 登入紀錄 / 登入 IP 追蹤 | V3 |
| OAuth / SSO / Google 登入 | 水深,不碰 |
| 多層級角色(總管理員 / 廠區管理員) | Dawn 的公司只有兩層:管理員跟設計師,夠了 |
| Audit Log(誰改了什麼) | V3 跟「可編輯提報」一起做 |
| 設計師個人看板 / 儀表板 | 清單頁就是看板,不用另外做 |

## 9. 怎樣算做完了(驗收標準)

V2 完成後,要跑通以下流程:

1. 管理員登入,進入帳號管理頁面
2. 新增兩個設計師帳號:designer_a / designer_b
3. 登出,用 designer_a 登入
4. designer_a 提報 3 筆款項,只看到自己的 3 筆
5. 登出,用 designer_b 登入
6. designer_b 看到空清單(0 筆),提報 2 筆,只看到自己的 2 筆
7. designer_b 在瀏覽器 URL 欄位直接打 designer_a 的提報 API,**系統回 403 不給看**
8. designer_b 試圖刪除 designer_a 的提報,**系統回 403 不讓刪**
9. 登出,管理員登入,看到全部 5 筆(含提報人欄位)
10. 管理員匯出 Excel,打開看到全部 5 筆 + 提報人欄位 + 「按提報人彙總」區塊
11. 管理員停用 designer_a 帳號
12. 用 designer_a 帳號嘗試登入,**登入失敗**
13. designer_a 的歷史提報資料仍然在報表裡(停用帳號不等於刪除資料)

**第 7、8 步是整份 V2 最重要的驗收——不是測功能,是測安全。**

---

## 10. 技術提醒

- **密碼:** 用 `bcrypt` 或 `werkzeug.security.generate_password_hash`,絕對不存明碼
- **Session:** Flask-Login 或自己寫 session middleware,存 user_id + role
- **每個 API 都要驗身份:** 用 decorator 包起來,例如 `@login_required` + `@role_required('admin')`
- **測試:** V2 的測試重點不是功能,是權限邊界——要故意用設計師帳號去打管理員 API,確認被擋

---

## 11. 開發順序建議

建議按這個順序做,每一步做完都是一個可驗收的里程碑:

1. **先做 users 表 + 登入/登出**——做完之後系統有「誰在用」的概念
2. **再改 reports 表加 user_id**——做完之後每筆提報知道是誰輸入的
3. **加設計師端的資料過濾**——做完之後設計師只看到自己的
4. **加管理員端的全域視圖**——做完之後管理員看得到全部
5. **做帳號管理頁面**——做完之後可以新增/停用設計師
6. **改 Excel 報表加「提報人」欄位**——做完之後報表完整
7. **跑一遍完整驗收測試**(第 9 節那 13 步)

每一步預估 30-60 分鐘,整個 V2 大約半天到一天。
